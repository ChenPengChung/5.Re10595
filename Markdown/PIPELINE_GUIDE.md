# Regrid Pipeline Guide

This project binds the regrid workflow to these two phase folders:

- `phase1_generategrid/`
- `phase2_generatecheckpoint/`

When a run starts and the following inputs exist, the workflow must be driven through `./run.sh` or `./run`, not by calling the Python scripts manually:

- `phase1_generategrid/oldgrid_I257_J129_g2.0_a0.5.dat`
- `phase1_generategrid/newgrid_I257_J129_g2.5_a0.5.dat`
- exactly one origin checkpoint under `phase2_generatecheckpoint/oldcheckpoint_*/`, `phase2_generatecheckpoint/step_*_origin*/`, or `restart/step_*_origin*/`

## Entry Points

Use this for the real pipeline plus job submission:

```bash
./run.sh --no-queue-check
```

Use this to verify the pipeline without compiling, submitting, or writing a
new checkpoint when no normal `restart/checkpoint/` is present:

```bash
./run.sh --preflight-only --no-queue-check
```

If a normal `restart/checkpoint/` already exists and the goal is specifically
to verify the phase1/phase2 regrid path, make the intent explicit:

```bash
./run.sh --regrid-from-origin --preflight-only --no-queue-check
```

Use this when you explicitly want to rebuild the regrid checkpoint from the origin and then submit:

```bash
./run.sh --regrid-from-origin --force-regrid --no-queue-check
```

`./run.sh` is a root-level compatibility wrapper. The implementation remains in `./run` and `chain_code/run.sh`.

## Automatic Workflow

The automatic path is:

1. Check that `phase1_generategrid/` contains exactly one matching `oldgrid_*.dat` and one matching `newgrid_*.dat` for `variables.h` `NY/NZ`.
2. Run `python3 J_Frohlich/grid_zeta_tool.py --auto` to ensure the solver-side grid exists.
3. Run `python3 phase2_generatecheckpoint/interp_checkpoint.py --auto --step 1` with the phase1 old/new grids and the unique origin checkpoint.
4. Compare the phase1 NEW grid against the exact solver grid derived from `variables.h` `GRID_DAT_DIR`, `GRID_DAT_REF`, `NY`, `NZ`, `GAMMA`, and `ALPHA`.
5. Require `restart/checkpoint/step_00000001/metadata.dat`, `f00_0.bin`, `rho_0.bin`, and `restart/grid_provenance`.
6. Enter Scenario 2 restart submission; the solver reads the generated checkpoint.

Do not pre-generate the checkpoint or copy grids by hand when testing the pipeline. The test is valid only when `./run.sh` triggers the generation itself.

## Agent Rules

- If a task touches `phase1_generategrid/` or `phase2_generatecheckpoint/`, check this guide first.
- Do not rename phase1 grids into `J_Frohlich/`; the solver grid must be generated by `J_Frohlich/grid_zeta_tool.py --auto`.
- Do not call `interp_checkpoint.py` directly for production submission; use `./run.sh --regrid-from-origin` only when forcing an explicit rebuild.
- If multiple old/new phase1 grids or multiple origin checkpoints exist, stop and make the selection explicit.
- For job safety, submit only through `./run.sh`, `./run`, or `./run build`; never bypass the project job guard for cancellation.
