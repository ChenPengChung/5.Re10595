計畫守門員 : paetition 數量以及 partition轉換計畫 
 Plan — Net-throughput-optimal, never-idle, jp×partition auto-switching daemon

 Context (why)

 The sim keeps stalling and wasting compute: the h200/gb200 federated partitions were
 removed, the old dispatcher (chain_code/submit_dispatcher.sh) loops "no-capacity" on the
 defunct candidates, and a jp=64 job is permanently rejected (MaxGRESPerAccount: per-account
 GPU cap = 32 on normal/4nodes). We already switched to jp=32 (checkpoint step_9500001,
 repartitioned + metadata fixed, validates, ready to resume).

 The user's true objective (confirmed): never let the run idle, and at every decision point
 automatically run the (jp × partition) combination with the highest net throughput — not
 the largest GPU count for its own sake. jp must be freely switchable, which requires making
 checkpoint repartition statistics-preserving (currently it drops the 36 sum_* fields).

 Hard facts that shape the design

 - Partitions (single H200 cluster): normal (2-day, cap 32 GPU/acct), 4nodes (1-day, cap 32),
 dev (1-hour, uncapped), large/slinky (gov-only → unusable).
 - H200 = 8 GPU/node. Valid jp for NY=641: 640%jp==0 && 640/jp≥7 && jp%8==0 → {16, 32, 64}
 (128 impossible: slab 5<7). **jp=64 ⇒ dev-only** (64>32 cap); jp≤32 fits all three.
 - jp is compile-time (variables.h #define jp) → each jp needs its own binary.
 - Repartition is bit-exact (pure per-rank slab stitch→split; y-axis is periodic, main.cu:426)
 — verified the 36 sum_* fields share rho's exact layout (all 14,542,344 B/file).
 - Stats not needed yet: FTT_STATS_START=40, current FTT=13.97, accu_count=0. So jp-switching
 is already lossless today → ship the daemon first, land stats-preservation before FTT 40.
 - Solver gates: [G6] checks mpi_rank_count+grid_dims (fileIO.h:~609); run.sh Preflight-C
 checks variables_h_mtime (NOT new_jp/new_chunk_j — those are cosmetic).

 Objective / decision logic

 Trigger points: (a) every round-end, (b) job stuck PENDING > timeout (~10 min).

 1. Read FTT + current jp (checkpoint metadata mpi_rank_count) + accu_count.
 2. Enumerate ADMISSIBLE combos c=(jp,partition):
      valid jp {16,32,64} × {normal,4nodes,dev}
      HARD filters: jp≤32 for normal/4nodes (cap); a.out.jp<N> exists; partition auth ok.
 3. For each c: probe start-ETA via `sbatch --test-only ... --exclude=25a-hgpn207`.
 4. Score net throughput (additive FTT-over-horizon, below); pick argmax.
 5. Apply: partition-only change → sbatch CLI flags (no file mutation).
           jp change → stop cleanly → repartition → cp a.out.jp<N> → update variables.h+provenance → submit.
 6. PENDING re-select: if submitted job pends > timeout → (race-guard) re-run selection (may
    pick a smaller jp / dev to start now). NEVER idle-wait while dev can place a job.

 Net-throughput model (corrected: additive, in FTT units)

 net(c) = r_ftt(jp) · max(0, H − startdelay(c) − switch_overhead(c))
 - r_ftt(jp) = realized FTT/hour at jp, measured from timing_log.dat (ΔFTT/ΔWall_min×60),
 persisted to restart/throughput_by_jp.dat. Cold-start: anchor on measured jp=32, provisional
 r_ftt(jp)=r_ftt(32)·(jp/32)^0.85 (sub-linear), replaced by measurement after one round.
 - startdelay(c) = probed ETA wait (hours; ≈0 for dev since uncapped).
 - switch_overhead(c) = repartition (~3 min, only if jp changes) + restart init.
 - H = 48 h at round-end; 1 h at PENDING re-select (favors start-now).
 - Never-idle by construction: dev always yields a placeable-now combo; selector returns a
 placeable combo unless a soon-free one strictly beats "dev-now + re-decide."
 - Anti-thrash: switch jp only if net(best_other_jp) > net(current_jp)·1.15; min 2-round
 dwell per jp; partition-only switches are cheap (margin ≈1.0).
 - Productivity guard: treat a combo as productive only if r_ftt(jp)·walltime ≥ one checkpoint interval (NDTBIN=100000); a dev slot too short
 to reach a checkpoint is net-negative.

 Codex review — folded-in corrections

 - P1 (critical race): jp-switch holds HEAD.lockdir in SUBMITTING for ~3 min, but HEAD_STALE_TIMEOUT=30s → another submitter would steal it and
 double-submit. Fix = new REPARTITIONING state (or scoped 600 s timeout) in head_lock_lib.sh, live until repartition completes. Must land
 before Phase 4.
 - P1 (my plan was wrong): the +7/+1 width formulas are algebraically equal (with-ghost vs without-ghost) → drop the "unify" rewrite, document
 instead. Keeps risk out of the stats code.
 - P2: mpirun -np 32 is hardcoded (not $NP) — the CLI-sizing design only works after this one-line fix. Confirmed correctness, not cleanup.
 - P3 (never-idle): productivity guard r_ftt·walltime ≥ checkpoint interval could make a 1-h dev slot net-negative and stall. Use NDTBIN/2 for
 dev (keep-alive path) so dev never gets fully excluded.
 - M1: PENDING re-select MUST cancel the stuck job via ./run job-guard scancel (never bare scancel) + re-check squeue/lock state to abort if it
 flipped PENDING→RUNNING in the race window.
 - M2: define a.out.jp<N> naming + presence check + a tiny manifest (jp↔binary md5) so the daemon never cps a mismatched binary.
 - M3: stats round-trip test before Phase 5 = generate mock sum_*_<rank>.bin with known values → repartition → round-trip → sha256 compare
 (real sum_* only exist after FTT 40).

 - Byte-exact round-trip gate before enabling any jp-switch: sha256 every f*/rho/sum_*
 after jp→X→jp must be identical. On ANY mismatch → freeze jp (partition-only switching) +
 write restart/STOP_JPSWITCH + loud log.
 - No nvcc in hot path (cp pre-built binary only); binary jp verified via manifest.
 - Never rebuild while a job runs; jp-switch only happens between rounds (no active job).

 - PENDING→RUNNING race guard: re-check squeue/lock state before any re-select scancel.
 - Repartition failure → fall back to current jp (submit, never idle); keep _bak until a clean
 new-jp checkpoint exists; retain only newest 1–2 _bak.
